คู่มือการใช้งาน (User Guide)

วิธีใช้งาน Web Confluent Kafka User Management แบบทีละขั้นตอน ให้ทำตามลำดับเพื่อลดข้อผิดพลาด

บทนำ

แอปนี้ใช้จัดการ User ของ Kafka บน Confluent (OCP CWDC + TLS2) ได้แก่ สร้าง topic ใหม่และ onboard user, เพิ่ม user ใหม่, เพิ่มสิทธิ์ (ACL) ให้ user ที่มีอยู่แล้ว, ทดสอบ user, ลบ user และ ACL, เปลี่ยนรหัสผ่าน, และ ลบ ACL ที่ค้างอยู่ หลังดำเนินการแต่ละอย่าง ระบบจะ validate ก่อนทำจริง และไม่ commit แบบครึ่งเดียว (ถ้าล้มกลางทางจะ rollback)

1. Add new user (เพิ่ม user ใหม่)

ใช้เมื่อต้องการสร้าง Kafka user ใหม่ พร้อม ACL สำหรับ topic ที่กำหนด และได้ไฟล์ .enc สำหรับแจกให้ผู้ใช้ถอดรหัสแล้วใช้งาน

ขั้นตอนการทำ

1 Identification & Topic
กรอก Client System Name (ใช้เป็นชื่อโฟลเดอร์/ไฟล์ของ pack) และ Topic name ที่มีอยู่จริงในคลัสเตอร์
กดปุ่ม Load topic list เพื่อดึงรายการ topic จาก Kafka แล้วเลือกจากรายการหรือพิมพ์ชื่อ (แนะนำให้เลือกจาก list เพื่อกันพิมพ์ผิด)
2 Username
กรอก Username ของ Kafka (ตัวอักษร/ตัวเลข/จุด/ขีดล่าง/ขีดเท่านั้น ไม่ซ้ำกับที่มีอยู่)
กดปุ่ม Load existing users เพื่อดูรายชื่อ user ที่มีอยู่ — ห้ามใช้ชื่อที่อยู่ใน list นี้
3 Permission (ACL)
เลือกสิทธิ์ของ topic: Read (consume อย่างเดียว) หรือ All (อ่าน+เขียน)
4 Passphrase
ตั้งรหัสผ่านสำหรับเข้ารหัสไฟล์ .enc (ต้องกรอกสองช่องให้ตรงกัน) — ใช้รหัสนี้เมื่อถอดรหัสไฟล์ .enc
5 Execute
ตรวจสอบข้อมูลใน Summary แล้วกดปุ่ม Create
รอจนเสร็จ จะมีข้อความสำเร็จ และลิงก์ Download ไฟล์ .enc พร้อมคำสั่งถอดรหัส (openssl) ด้านล่าง
หลัง Add สำเร็จ ระบบจะรันการทดสอบ auth ให้อัตโนมัติ และแสดงผล Verification ว่าผ่านหรือไม่
ถ้า Username มีอยู่แล้ว: ระบบจะแสดงข้อความ "Username already exists" พร้อมลิงก์ add this topic to this user — กดลิงก์เพื่อไปหน้า Add ACL to existing user โดยระบบจะเติม Username และ Topic ให้อัตโนมัติ
หลังดาวน์โหลด: ใช้คำสั่งในบล็อก "How to decrypt and unpack" เพื่อถอดรหัสไฟล์ .enc (ใส่ passphrase ที่ตั้งไว้) จากนั้นแตกไฟล์และส่งโฟลเดอร์ (หรือ client.properties + credentials) ให้ผู้ใช้ Kafka

2. Create Topic & Onboard User (สร้าง topic ใหม่และ onboard user)

ใช้เมื่อต้องการสร้าง Kafka topic ใหม่จาก Web แล้ว onboard user คนแรกสำหรับ topic นั้นใน flow เดียว (ขั้นที่ 1 สร้าง topic, ขั้นที่ 2–6 เหมือน Add new user: Client System Name, Username, ACL, Passphrase, Execute)

ขั้นตอนการทำ

1 Create Topic — กรอกชื่อ topic ที่ต้องการสร้าง แล้วกดปุ่มสร้าง topic (ใช้ค่า default ของ broker สำหรับ partitions และ replication factor)
2 Identification & Topic — กรอก Client System Name และ topic จะถูก pre-fill จากขั้นที่ 1
3–6 Username, Permission, Passphrase, Execute — ทำเหมือน Add new user (รวมถึงการโหลดรายการ users และเลือก ACL)
ถ้าใส่ Username ที่มีอยู่แล้ว ระบบจะแจ้งและมีลิงก์ไป Add ACL to existing user เพื่อเพิ่ม topic นี้ให้ user นั้น

3. Add ACL to existing user (เพิ่มสิทธิ์ให้ user ที่มีอยู่แล้ว)

ใช้เมื่อ user มีอยู่แล้วในระบบ และต้องการเพิ่มสิทธิ์ (ACL) สำหรับ topic อื่น โดยไม่สร้าง user ใหม่หรือเปลี่ยนรหัสผ่าน — เพิ่มเฉพาะ topic ACL และ consumer group ACL ให้ user นั้น

ขั้นตอนการทำ

1 กดปุ่ม Load users และ Load topics เพื่อดึงรายการล่าสุด
2 กรอก Username (user ที่มีอยู่แล้ว) และ Topic ที่ต้องการเพิ่มสิทธิ์ — หรือคลิกเลือกจากรายการ
3 กดปุ่ม Load existing ACLs เพื่อดูสรุปสิทธิ์ปัจจุบันของ user นั้น
4 เลือกระดับสิทธิ์ (Read / Client (R+W+Describe) / All) แล้วกด Prepare — ระบบจะแสดงข้อความยืนยัน
5 กด Confirm เพื่อดำเนินการ — หลังสำเร็จการกระทำจะถูกบันทึกใน Audit Log เป็น "Add ACL (existing user)"

4. Test existing user (ทดสอบ user ที่มีอยู่)

ใช้ตรวจสอบว่า user สามารถเชื่อมต่อ Kafka ได้ (auth ผ่านและเข้าถึง topic ได้)

ขั้นตอนการทำ

1 กดปุ่ม Load topics และ Load users ในส่วน Test existing user เพื่อดึงรายการล่าสุด
2 กรอก Kafka username, Password (รหัสที่ได้จากไฟล์ที่ถอดรหัสแล้ว เช่น ใน credentials.txt หรือ client.properties), และ Topic name
สามารถคลิกเลือกจากรายการ topic/user ที่โหลดไว้ได้
3 กดปุ่ม Test — ถ้าสำเร็จจะแสดง "Auth OK." ถ้าไม่จะแสดงข้อความ error และรายละเอียด (Details) สำหรับใช้ trace

5. Remove user(s) + ACL (ลบ user และ ACL)

ใช้เมื่อต้องการลบ user ออกจากระบบ (ลบ ACL ทั้งหมดของ user นั้นก่อน จากนั้นลบออกจาก secret CWDC และ TLS2) ระบบจะ validate ก่อนว่าอ่าน secret ได้และ user มีอยู่ในทั้งสองฝั่ง จึงจะลงมือลบ และถ้าล้มกลางทางจะไม่ทิ้งให้แก้แค่ฝั่งเดียว (rollback)

ขั้นตอนการทำ

1 กดปุ่ม Load existing users ในส่วน Remove user(s) + ACL เพื่อดึงรายการ user ล่าสุด
2 ในช่อง Users (comma-separated) กรอกชื่อ user ที่ต้องการลบ (หลายคนคั่นด้วย comma) หรือคลิกเลือกจากรายการที่โหลดไว้
3 กดปุ่ม Remove user(s) and ACL
ระบบจะ: (1) ตรวจสอบว่าอ่าน secret CWDC/TLS2 ได้ และ user มีในทั้งสองฝั่ง (2) ลบ ACL ทั้งหมดของ user เหล่านั้น (3) ลบ user ออกจาก secret ทั้งสองฝั่ง — ถ้าขั้นใดขั้นหนึ่งล้มเหลวจะไม่ commit แบบครึ่งเดียว
คำเตือน: การลบ user จะทำให้ user นั้นเข้า Kafka ไม่ได้หลังจาก CFK hot-reload (~30–60 วินาที) แนะนำให้แจ้งผู้ใช้ก่อนลบ

6. Change password (เปลี่ยนรหัสผ่าน)

ใช้เมื่อต้องการเปลี่ยนรหัสผ่านของ user ที่มีอยู่แล้ว ระบบจะ validate ว่า user มีใน secret ทั้ง CWDC และ TLS2 ก่อน จากนั้นอัปเดตทั้งสองฝั่ง หรือ rollback ถ้าล้มกลางทาง

ขั้นตอนการทำ

1 กดปุ่ม Load users ในส่วน Change password เพื่อดึงรายการ user
2 กรอก Username ของ user ที่จะเปลี่ยนรหัส (แนะนำให้เลือกจาก list) และ New password ที่ต้องการตั้ง
3 กดปุ่ม Change password
หลังเปลี่ยนแล้ว รหัสเก่าจะใช้ไม่ได้หลัง CFK reload — แจ้งผู้ใช้ให้ใช้รหัสใหม่และอัปเดต client.properties/credentials

7. Cleanup orphaned ACLs (ลบ ACL ที่ค้างอยู่)

ใช้เมื่อมี principal ใน ACL ของ Kafka ที่ไม่มีอยู่ใน plain-users.json แล้ว (เช่น ลบ user ไปแล้วแต่ ACL ยังค้าง) ระบบจะ list ACL ทั้งหมด แล้วลบเฉพาะรายการที่ principal ไม่ได้อยู่ใน secret — ไม่มีการแก้ secret แค่ลบ ACL ใน Kafka

ขั้นตอนการทำ

1 ไปที่ส่วน Cleanup orphaned ACLs แล้วกดปุ่ม Run cleanup
2 รอจนเสร็จ จะแสดงจำนวน ACL ที่ลบออก (ถ้ามี) หรือข้อความสำเร็จ

8. Audit Log & Download History

Audit Log — แสดงประวัติการดำเนินการ (Add user, Create topic, Add ACL existing, Remove user, Change password, Test user, Cleanup ACL) สามารถกรองตามช่วงวันที่ (From / To รูปแบบ YYYY-MM-DD) แล้วกด Refresh

Download History — แสดงรายการวันที่มีการ generate credential pack พร้อมลิงก์ดาวน์โหลดไฟล์ .enc ตามวัน

เมื่อเกิดข้อผิดพลาด

ถ้าแอปแสดงข้อความ error (สีแดง):

  • อ่านข้อความหลักด้านบน — มักบอกสาเหตุสั้นๆ (เช่น gen.sh exited 1, config not found)
  • ถ้ามี Step / Phase / Exit code ด้านล่าง ใช้สำหรับอ้างอิงเวลาแจ้ง Admin หรือดู log
  • กดเปิด Details (stderr) หรือ Details (stdout) เพื่อดูข้อความจากสคริปต์/คำสั่ง — ใช้ trace ปัญหาใน production ได้

Admin สามารถดู log ที่ server (ข้อความขึ้นต้นด้วย [add-user], [remove-user] ฯลฯ) และที่เครื่องที่รัน gen.sh ดูที่ไฟล์ provisioning.log (path ตาม config) เพื่อดู step และ exit_code เพิ่มเติม

Release Notes

สรุปการเปลี่ยนแปลงและฟีเจอร์หลักในแต่ละเวอร์ชัน (อัปเดตตาม app version ใน Web)

1.0.44

  • UI: เปลี่ยน label "System name" เป็น "Client System Name" ทั้งใน Add new user และ Create Topic & Onboard เพื่อสื่อความหมายฝั่ง client ให้ชัดเจน
  • UI: เมื่อกรอก Username ที่มีอยู่แล้ว (ใน Add user หรือ Create Topic & Onboard) จะแสดงลิงก์ "add this topic to this user" — กดแล้วไปหน้า Add ACL to existing user พร้อม pre-fill Username และ Topic
  • Docs: อัปเดต Documentation ให้รวม Create Topic & Onboard, Add ACL to existing user, Audit Log & Download History และ Release Notes

1.0.43

  • UI: Audit Log ช่องวันที่ From/To ใช้ placeholder "YYYY-MM-DD" (ภาษาอังกฤษ) แทนรูปแบบภาษาไทย

1.0.42

  • ฟีเจอร์: Add ACL to existing user — เพิ่มสิทธิ์ (topic + consumer group) ให้ user ที่มีอยู่แล้วโดยไม่สร้าง credential ใหม่
  • Web: หน้า "Add ACL to existing user" — โหลด users/topics, แสดงสรุป ACL เดิม, เลือกระดับสิทธิ์, ยืนยันแล้ว Execute
  • Audit: บันทึกการสร้าง topic (create-topic) และ Add ACL existing (add-acl-existing) ใน Audit Log พร้อม label อ่านง่าย
  • CLI: gen.sh โหมด [5] Add ACL for existing user (ทั้งแบบ interactive และ non-interactive GEN_MODE=5)

1.0.41 และก่อนหน้า

  • Create Topic จาก Web (GEN_MODE=4), Add user, Test user, Remove user, Change password, Cleanup ACL
  • Login พร้อม security code (เมื่อเปิด auth), session, favicon 204, /api/me 200 เมื่อยังไม่ login
  • แก้ไข syntax error ใน Create Topic/Add user error message และการโหลด security code หลัง auth mode
